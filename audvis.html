Overview

Sound you can see, built entirely in the browser.

AudVis started as a challenge to myself: could I build something visually impressive using only browser-native APIs, zero libraries, and zero canvas frameworks? The answer was yes ‚Äî and the process taught me more about real-time graphics rendering and audio processing than any tutorial could.

The result is an interactive audio visualizer that accepts sound from three different sources, transforms raw frequency data via FFT analysis, and renders it in real time across four distinct visualization modes ‚Äî all running at smooth framerates in a single HTML file.

The Challenge

Real-time graphics and audio processing are unforgiving.

Canvas animations have no safety net. A poorly optimized render loop drops frames immediately and visibly. Combining that with live audio analysis ‚Äî where the data updates 60 times per second ‚Äî meant every decision had a direct performance cost.

Audio pipeline complexity

Each input source ‚Äî microphone, file, stream ‚Äî needs its own routing through the Web Audio API graph. Building a unified interface over three different source types without redundant code was the first real design challenge.

Canvas performance

Clearing and redrawing the entire canvas every frame is expensive. I had to learn which operations to batch, when to use requestAnimationFrame vs setInterval, and how to avoid unnecessary state changes.

Technical Architecture

Web Audio API does the heavy lifting.

The Web Audio API exposes an AnalyserNode that performs Fast Fourier Transform (FFT) analysis on live audio, returning frequency-domain data as a byte array. That array is what drives every visual ‚Äî each bar, wave, and particle maps directly to a frequency bucket.

Web Audio API
Core audio graph: source nodes feed into an AnalyserNode that extracts frequency and waveform data at configurable FFT sizes. Higher FFT = more frequency resolution, higher CPU cost.

HTML5 Canvas 2D
All visuals rendered via the Canvas 2D context. requestAnimationFrame drives the loop, keeping rendering in sync with the display refresh rate without manual timing logic.

getUserMedia() captures microphone input as a MediaStream, which is then routed into the Web Audio graph ‚Äî the same pipeline as a file or stream source.

Zero external libraries. Every animation, every control, every audio routing decision is hand-written. The constraint made me understand the APIs far more deeply than abstracted equivalents would have.

Visualization Modes

Four ways to see the same sound.

Each mode uses the same underlying frequency data but visualizes it with a completely different algorithm. Building each one from scratch deepened my understanding of the data itself ‚Äî how frequency buckets cluster, how amplitude maps to perceived loudness, how time-domain vs frequency-domain data behaves.

What I Learned

Performance is a design decision.

This project changed how I think about front-end work. Every line of code in a render loop has a cost that shows up visibly and immediately. That feedback loop ‚Äî write something, see it stutter, figure out why ‚Äî made performance optimization feel less like an optimization and more like a design constraint I had to work within from the start.

I also developed a much deeper appreciation for what browser APIs can actually do. The Web Audio API's node graph architecture is genuinely elegant ‚Äî routing audio through composable nodes mirrors how physical signal chains work, which made understanding it feel intuitive once it clicked.

Building with browser-native APIs means understanding exactly what the platform can do ‚Äî not just what a framework exposes.
